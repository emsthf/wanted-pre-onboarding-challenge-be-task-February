1. 관계형 데이터베이스(RDBMS)와 비관계형 데이터베이스(NoSQL)의 장단점 비교

관게형 데이터베이스는 데이터베이스 유형 중 하나로 구조화가 특징인 데이터베이스이다. 데이터들 사이의 관계성을 가지는 데이터들을 다루며, 구조적으로 데이터를 저장한다.
2차원 구조 모델(열 : 속성, 행 : 튜플, 데이터)을 기반으로한 수평적 구조를 사용, 테이블 형식으로 데이터를 저장하고 각 테이블들은 많은 행과 열로 구성되며, 테이블은 구조화된 데이터를 저장하기 위한 엄격한 구조를 제공한다.
ACID(원자성, 일관성, 독립성, 지속성) 트랜잭션을 보장하는 기능을 가지는데, 이 기능들은 모든 트랜잭션이 완전하고 정의된 모든 규칙에 따라서 일관성을 가지게 함으로써 오류, 시스템 충돌과 같은 문제가 없는지 확인할 수 있도록 한다. 이로써 데이터 무결성, 완전성, 정확성을 보장할 수 있도록 하는 것이다.
SQL을 사용해서 데이터 접근을 하며 CRUD 조작을 허용하며 대표적인 RDBMS로는 MySQL, Oracle, MariaDB 등이 있다.





2. 트랜잭션(transaction)이란 무엇인가요?

트랜잭션이란 데이터베이스의 상태를 변화시키기 위해 수행하는 작업 단위를 말한다. 즉, 하나의 트랜잭션 설계를 잘 만드는 것이 데이터를 다룰 때 많은 이점을 가져다준다.
트랜잭션의 특징으로는 위에 언급한 ACID(원자성, 일관성, 독립성, 지속성)이 있으며 commit으로 하나의 트랜잭션이 성공적으로 끝났고, 변경된 데이터를 영구적으로 반영할 수 있다.
rollback은 하나의 트랜잭션 처리가 비정상적으로 종료되어 트랜잭션 원자성이 깨진 경우 데이터 변경사항이 취소되어 이전 상태로 되돌릴 수 있게하고 관련 데이터의 lock이 풀려 다른 사용자들이 데이터를 변경할 수 있도록 한다. 세이브포인트를 정의하면 롤백할 때 트랜잭션의 모든 작업을 롤백하는 것이 아닌 현 시점에서 세이브포인트까지만 롤백이 되도록 할 수 있다.




3. MySQL에서 조인(join)의 역할은 무엇인가요? 다양한 join의 방식에 대해 설명해주세요.

조인이란 데이터베이스 내에서 2개 이상의 테이블을 합쳐 SELECT를 하는 방법 중 하나이다.
조인에는 다양한 종류가 있는데 inner join은 교집합을, outer join은 교집합 이외에 테이블의 조건에 맞는 모든 부분을 말한다. outer join에서 left, right를 기준으로 어느쪽 테이블을 가져올지 정할 수 있다. self join은 말그대로 자기 자신을 Join하는 방법이다. A 테이블을 각각 Child, Parent로 지정해서 사용한다.




4. MySQL에서 인덱스(index)란 무엇인가요?

인덱스란 테이블의 동작속도(조회)를 높여주는 자료구조이다. 인덱스로 데이터의 위치를 빠르게 찾아주는 역할이고 쉽게 예를 들어보면 책 뒷편에 '색인'이 인덱스의 역할과 동일하다고 볼 수 있다. 
인덱스는 MYI(MySQL Index)파일에 저장되며, 인덱스가 설정되지 않았다면 Table Full Scan이 일어나 성능이 저하되거나 치명적인 장애가 발생한다. 조회속도는 빨라지지만 UPDATE, INSERT, DELETE의 속도는 저하된다는 단점이 있다. 그래서 효율적인 인덱스 설계로 단점을 최대한 보완해야 한다.
인덱스는 하나 혹은 여러 개의 컬럼에 대해 설정할 수 있다. 
다중 컬럼 인덱스는 단일 컬럼 인덱스 보다 더 비효율적으로 INDEX/UPDATE/DELETE를 수행하기 때문에 신중해야한다. 때문에 가급적 UPDATE가 안되는 값을 선정해야한다.
WHERE절을 사용하지 않고 인덱스가 걸린 컬럼을 조회하는 것은 성능에 아무런 영향이 없다
INDEX는 ORDER BY와 GROUP BY에도 영향을 끼치는데 다음과 같은 경우에는 INDEX를 타지 않는다.
ORDER BY 인덱스컬럼1, 컬럼2 : 복수의 키에 대해서 ORDER BY를 사용한 경우
WHERE 컬럼1='값' ORDER BY 인덱스 컬럼 : 연속하지 않은 컬럼에 대해 ORDER BY를 실행한 경우
ORDER BY 인덱스컬럼1 DESC, 인덱스컬럼2 ASC : DESC와 ASC를 혼합해서 사용한 경우
GROUP BY 컬럼1 ORDER BY 컬럼2 : GROUP BY와 ORDER BY의 컬럼이 다른 경우
ORDER BY ABS(컬럼) : ORDER BY 절에 다른 표현을 사용한 경우

추천하는 인덱스 설계 방법으로는
무조건 많이 설정하지 않는다. (한 테이블당 3~5개가 적당 목적에 따라 상이)
조회시 자주 사용하는 컬럼
고유한 값 위주로 설계
카디널리티가 높을 수록 좋다 (= 한 컬럼이 갖고 있는 중복의 정도가 낮을 수록 좋다.)
INDEX 키의 크기는 되도록 작게 설계
PK, JOIN의 연결고리가 되는 컬럼
단일 인덱스 여러 개 보다 다중 컬럼 INDEX 생성 고려
UPDATE가 빈번하지 않은 컬럼
JOIN시 자주 사용하는 컬럼
INDEX를 생성할 때 가장 효율적인 자료형은 정수형 자료(가변적 데이터는 비효율적)
